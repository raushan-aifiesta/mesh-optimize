2/
the insight nobody believes until they do the math:

your IDE agent re-sends a ~120k token system prompt on EVERY request. plus the whole conversation. the api is stateless.

50 tool calls = the same bytes billed 50 times at full price.

most of your bill is repetition.

3/
"but caching only works for identical prompts"

wrong, and this misconception is the product.

caching is a PREFIX match. tools + system prompt + old turns are identical bytes. only your newest turn changes.

cached read on fable: 10% of input price. $1.20 becomes $0.12. per request.

[attach: prefix-match.svg]

4/
so: one request param. mesh_optimize, 0 to 0.95.

0.3 gets you cache breakpoint injection, max_tokens backstops, and pruning of tool outputs the model already consumed.

every response carries a receipt:

tokens_saved: 3488
cost_saved_usd: 0.9349
levers_applied: [...]

real output, not a mock.

[attach: try.ts terminal screenshot]

5/
the rule that surprised me most while building:

NEVER rewrite content inside a cache breakpoint to save tokens. cached bytes cost 10%. fewer bytes at full price loses to more bytes at 10%.

cheaper bytes beat fewer bytes. tattoo it somewhere.

6/
honesty constraints we hard-coded:

- savings only claimed when measurable
- estimation method disclosed in every receipt
- cold cache writes can make savings NEGATIVE and we report that too. there's a unit test asserting we don't clamp it.

an inflated savings number is an ad. ours is a receipt.

7/
our first live request returned HTTP 500.

the baseline. zero optimization. the bug was my mental model of our own gateway: no top level system field, tool messages need tool_call_id, model ids are anthropic/claude-opus-4.8 not claude-opus-4-8.

reading your own api docs is underrated.

[attach: 500 error terminal screenshot]

8/

- cache lever now handles openai-shape system messages (how traffic actually arrives)

failures in week one beat support tickets in month six.
